    CheckActivityItemAssociation 也同时考虑，是否作为一张模块内聚的业务表，还是切换到data-association

data-association（下面简称da）
背景：
超音速中存在大量的数据关联需求，基于将来数据查询优化、数据扁平化的设计思路，不使用多对多查询，而是使用关联表；
data-association存在的主要目的在于解决超音速中多处的数据关联需求，避免每个模块都重复造轮子。

序号 方案 优点 缺点
1 da作为统一服务，所有数据关联都使用该服务 高效，不需要每个模块重复编写数据关联代码 模块不够内聚，业务字段查询效率比较低；da引入服务复杂度
2 每个模块独立维护da 模块功能内聚，简单 da功能在多个模块中重复实现和维护，开发效率低
3 da作为功能插件，每个模块可以使用 模块内聚，开发效率开 功能插件的形式待预研

建议选择方案3

异常处理

  • 网络异常、url异常等，这类异常用户无法处理，也不需要系统编程错误，只需要登记log，直接返回执行失败的标志即可，不需要一层层往上抛出错误；因为通常上层也不会处理；基于系统健壮性考虑，可以尝试进行重试机制

思考：
流程节点的数据、参数、配置，应该归属到流程节点配置、还是在工具做配置、还是在产品做配置

超音速平台

发表于 2017-02-05

一，系统的业务边界（哪些有系统做）

1. 研发活动

研发活动天然的围绕着代码配置管理工作，包括代码的分支、tag、推拉更新；所以在研发
从开发->测试->发布这中间的环节中，更多的是需要对代码活动可以有监控。

对代码不同分支的不同活动监听，然后有不同的业务行为：例如，

  • 从个人分支回到主干分支，必须进行代码review；
  • 在开发分支，可以配置是否需要代码review
  • 回主干之后，配置进行哪些持续集成
  • 在开发完某个功能后的开发打包，或者回主干前的测试打包

总结主要包括的行为：

  • 代码走查
  • 持续集成
  • 打包

目的是：

  • 开发质量保障
  • app的自动化测试：性能、稳定性测试
  • 快速问题定位

研发环节涉及的业务：

  • 配置管理工具：svn、git > 提供接口，或者hooks
  • 超音速平台，对接口做封装？提供注册？
  • 机器可以使用jenkins做接口hook？(这样其实相等于平台做了一层中间层)

P.S. 其实也可以把这些接口的触发当作某种任务才有的特性：

  • 任务可以选择是否包含这些触发器，由/不由这些触发器触发
  • 任务可以包含前置任务和后置任务
  • 任务可以包含子任务
  • 任务可以形成一个任务集合

2. 流程

伴随着研发活动，需要进行流程的发起，场景如下；实际上流程复杂的是准入条件的控制以及重要活动的审批/知会/记录：

  • 灰度分支合并时，需要发起分支灰度流程
  • 合并到稳定分支时，需要发起发布流程

3. 关联系统

3.1. 简介
  • 代码配置管理：git/svn
  • 代码review：reviewboard，fhabricator，git merge request
  • 静态代码扫描：sonar，lint，coverity，firebug
  • 静态代码扫描（ios）：OClint，CLang，Faux Pas
  • 自动化测试：UCTC、QMS（iosQMS）(性能启动时间测试、稳定性测试、核心数据、网速测试、其他自动测试等)
  • 监控：崩溃日志系统、业务指标监控、线上系统监控、用户反馈
  • 发布：有爱平台、升级平台、下载平台
3.2 关系梳理

梳理数据流和交互关系。

  • git/svn：仓库地址/配置；代码（包/类/方法）；tag/分支branch；代码活动（推拉/分支branch拉取/tag创建/分支合并）等
  • 代码review/静态代码扫描：输入：代码，仓库地址/配置；输出：检查结果
  • 自动化测试：输入：开发/测试/内测/分支/灰度/正式包；输出：测试结果
  • 崩溃日志：用户/崩溃日志，输入：用户操作；输出：崩溃现场和代码错误日志
  • 用户反馈：用户/崩溃反馈，输入：场景说明；输出：用户/崩溃场景说明
  • 各类发布平台：用户/包

2016QCON分享

发表于 2017-02-02

讲什么：

1. 快速发展业务和技术架构／设计的关系
2. 他们优秀的设计思想和解决思路，以及（尝试总结）新趋势
3. 当下很火热的微服务

技术是业务和产品发展的产物：三个阶段

工程师常有的困惑：版本走的太快了，我们欠下不少技术债务，我们有不少技术优化点要做，技术需求优先级调了又调，排期delay。
这次看到的这些优秀的互联网公司，他们有没有这样的困扰，他们怎么思考这个问题。

业务／产品阶段：

1. 初创期：快速支持业务铺开
2. 快速爆发／成熟期：技术改进
3. 稳定期：生态化
不同阶段有不同的研发策略，核心是：技术为业务而生，业务驱动的改进。

架构演技和设计思路：三板斧

  1. 拆：解耦
    将变化的业务和通用的接入层分开
  2. stateless，无状态化
    价值：降低成本，易于水平扩展扩展
  3. 服务健康状态透明化（多盟，服务转换成可视化的业务价值）
  4. 一个观点：简单即是强大

微服务

who：豆瓣，小米广告平台，今日头条，etc
why：解决”肥胖病”，提高效率，更快的产品交付
亮点：
阿里：从服务的角度，和docker结合：容器即服务－container as a service

组件发布流程的简要设计

发表于 2017-02-02

app

  • constants
    meta data

    • FlowNodeType
      节点类型,目前为组件发布流程定制,包括DEV,TEST,LEADER这3个节点
    • FlowResult
      节点/流程结果,包括:初始状态,通过,失败,驳回
    • FlowStatus
      节点/流程状态,包括:初始状态,运行中,结束,取消

  • models.flow
    all about 数据

    • FlowForm
      流程表单的数据,记录整个流程的流转、状态和结果;行为有:启动,下一节点,结束,通过FlowFormEngine判断
    • FlowNode
      流程节点的数据,记录节点的流转、状态和结果;行为有:创建,结束,获取下一节点

  • service.flow
    all about 流程流转的通用逻辑

    • FlowFormService
      流程的启动,流转,查询服务接口

  • dto.flow
    all about 数据视图

    • adapter
      数据视图转换层
      • FlowFormAdapter
      • FlowNodeAdapter
    • FlowFormDto
      流程表单的数据视图
    • FlowNodeDto
      流程节点的数据视图

  • process
    all about 业务逻辑, 可能经常变动的

    • FlowFormEngine
      根据流程和业务逻辑,产生下一节点
    • FlowNodePostProcessor
      节点后置处理
    • FlowFormDataFetcher
      获取,解析flow表单的业务数据内容
    • FlowFormMailer
      组件发布流程的邮件通知

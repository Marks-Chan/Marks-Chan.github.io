摩天轮组件化平台建设沟通

发表于 2017-03-12

摩天轮

沟通框架

  • 整体项目运作模式，以及框架体系、背后的思路是什么
  • 发布流程是否标准化的，业务是否有定制
  • 发布场景和策略、规范
  • 哪些是可以作为公共服务的，以及对应如何做业务定制和适配
  • 研发、生产上的公司级别的规范

业务流程

新概念：项目区，版本，集成区

  • 创建产品的3种类型，创建项目的2种类型
    客户端，自动创建hotpatch（分支不同）
    模块可以关联到客户端

  • 组件依赖管理的逻辑，包括线下开发、线上测试、发布的场景，特别是组件集成发版的逻辑

  1. 开发时一般是独立开发，需要打包时到摩天轮上进行打包；接下来计划做手机开发时的热部署，把整理代码热部署到本地研发测试手机
  2. 集成、发布时，管理产物／二进制，不管理源代码
  3. 有代码权限，才可以添加源码产品依赖，外部依赖不限制
  4. 关联版本如何发布（A->B）
    创建A项目，添加A模块的源码依赖，添加B模块的版本依赖，应用线上／集成区的依赖，测试打包等
  5. snapshot版本不允许集成
  • 紧急集成、预集成、卡点的逻辑
  1. 集成区，用来对一个产品做集成发布（将N个模块／SDK的发布集成到一起发布）
  2. 在单个模块项目中，创建集成单，来制定集成到那个集成区，从而发布；
  3. 创建集成单的时候，会进行集成卡口，校验是否符合创建集成单的要求（例如静态代码检查等）
  4. 创建集成单之后，进行预集成：先拉集成区的依赖来打包，解决以前直接集成导致打不出包的问题
  5. 紧急集成，扣积分，定期做数据review
  • 发布流程
    有一套完成的规范并落地
  • 响应回归
    希望2个小时做完回归，但是不及时，就变成30分钟提醒一次，3次没有响应就升级
  • 适配回归怎么做的
    外包资源测试，以及真机测试
  • code review这些研发活动
    更多是研发管理，没有在摩天轮做落地
  • 测试工具的效率
    monkey大概30分钟，下一步有计划做 集成的检查流水线，自动做编译、构建、静态检查、monkey等
  • andfix和热部署同时存在，主要原因是andfix适用范围更广

  • 线上版本的回滚
    不建议做回滚，考虑模块依赖的复杂度；
    一般紧急停止全量，快速bugfix然后热部署
    灰度不做回滚

  • 渠道预装发布的管理
    只做一个包模版，改N个渠道号
    预装要支持，可以考虑在打包模版里面做，现在mtl上本身不支持这么定制化的需求
    渠道发布目前还是线下发

  • 发布之后，会把关联的需求设为“已发布“

  • 白名单
    产品发布策略，推拉结合
  • 卡口
    可以配置对性能、P1/P2的bug做卡点

技术

  • atlas框架的对接是怎么做的
    服务器上部署框架即可，通过脚本执行

  • 发布架构，包括如何接入新的产品以及和超音速打通
    update-sdk，需要客户端支持（最好组件化，否则推送包可能比较大）
    推拉结合

  • 测试工具的对接，是否有统一标准，以及业务接入方式
    mcip管理后台，定制接入，封装成统一服务给摩天轮调用

  • 机器和环境管理，以及技术架构，是否能够直接复用服务或技术
    管理后台mtlmc，100多台服务器，另外手机也会在这里

  • 文档和总结

  • mtl系统是否组件化的

  • 报表定制的组件化开发和自定义报表
    主要是贾乐负责

  • 产品配置，业务模块、流程定制、卡点配置
    暂无，将来做配置中心

  • 手机管理
    wwdm，技术上可以独立部署

  • 和架构组为同一大团队
